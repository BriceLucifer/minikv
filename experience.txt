minikv production and performance experience

Date: 2026-07-02

Scope
-----
Review the C++ rewrite for production readiness, compare it with
geohot/minikeyvalue, run functional and compatibility tests, investigate local
performance limits, and make conservative fixes that improve operability.

Machine
-------
- MacBook Pro Mac16,8
- Apple M4 Pro, 14 cores
- 24 GB RAM
- macOS 26.5.1 (25F80)

Validation
----------
The repo is not judged only by unit tests. The important local gates are:

- cmake --build --preset debug
- ctest --preset debug --output-on-failure
- cmake --build --preset release
- MINIKV_REQUIRE_HTTP_COMPAT_DEPS=1 ctest --preset debug -R UpstreamCompatTest
- MINIKV_REQUIRE_S3_COMPAT_DEPS=1 ctest --preset debug -R S3CompatTest
- clang-tidy -p build/debug $(rg --files -g '*.cpp' src tests) --quiet
- clangd --check on representative source files
- go run tools/thrasher.go -h

The deploy-style tests start real nginx/WebDAV volumes and one C++ master, so
they cover the storage topology that matters in production.

What failed first
-----------------
Early high-concurrency PUT+DELETE tests produced HTTP 500s and client socket
errors. The topology was:

  client -> C++ master -> nginx/WebDAV volumes

It was not an nginx proxy in front of C++. The first observed failure mode was
short-lived TCP churn: benchmark clients disabled keep-alive, the master opened
fresh volume connections, and the host accumulated many local sockets in
TIME_WAIT.

Useful check:

  netstat -an | rg 'TIME_WAIT|CLOSE_WAIT' | wc -l

On this machine the count reached roughly 24822 during bad runs, then returned
to single digits after cooldown. That made long short-connection benchmarks a
test of local TCP resources, not a clean service-capacity measurement.

Source findings
---------------
- The master is synchronous per accepted connection on a Boost.Asio thread
  pool.
- Request bodies for ordinary PUT are still materialized in memory up to
  MINIKV_MAX_BODY_SIZE.
- GET redirects probe volume replicas with HEAD before returning 302.
- The original volume client created a fresh HTTP connection per replica
  operation.
- macOS sample showed hot time in Boost.Beast/Asio socket read, connect,
  kevent, and volume HEAD/PUT/DELETE paths. LevelDB and MD5 were visible but
  not dominant.

What changed
------------
- Added a configurable worker count: MINIKV_WORKERS / -workers.
- Changed the automatic worker default to a keep-alive-aware minimum of 64.
- Added inbound HTTP/1.1 keep-alive with bounded requests and timeouts.
- Added ordinary master-to-volume keep-alive pooling:
  MINIKV_VOLUME_POOL / -volumepool.
- Shared one Asio io_context per volume endpoint pool to avoid fd pressure from
  one io_context per pooled connection.
- Kept MINIKV_PARALLEL_REPLICAS false by default; parallel replica writes can
  increase volume pressure and connection churn.
- Added replica failure logging with operation and remote URL.
- Added deploy templates, body limits, pinned fetched dependency revisions,
  S3-style XML errors, and production runbook guidance.
- Added tools/thrasher.go as a local clone of upstream's write/read/delete
  thrasher.

Benchmark evidence
------------------
Same machine, release builds, local nginx/WebDAV topology:

| Implementation | Missing GET | 10k write/read/delete |
| --- | ---: | ---: |
| C++ rewrite | 173,036 req/s | 1,800/sec |
| Go upstream 451d248 | 163,872 req/s | 1,099-1,191/sec |
| Previously reported upstream reference | 116,338 req/s | 3,815/sec |

The C++ rewrite is ahead of the same-machine Go baseline for the measured
workloads. The external 3,815/sec thrasher number was not reproduced locally,
so it should not be treated as an apples-to-apples target without matching
hardware, OS, nginx config, filesystem, and process limits.

Earlier keep-alive samples with 64 KiB objects and five local volumes:

- c16 missing GET: 9.55k RPS, p99 1.63 ms, 0 errors
- c16 GET redirect: 10.09k RPS, p99 0.76 ms, 0 errors
- c16 GET follow: 5.36k RPS, 334.8 MiB/s, p99 3.69 ms, 0 errors
- c16 PUT+DELETE: 1.66k RPS, 104.1 MiB/s, p99 13.79 ms, 0 errors
- c32 GET follow: 5.05k RPS, 315.7 MiB/s, p99 10.18 ms, 0 errors
- c32 PUT+DELETE: 1.54k RPS, 96.1 MiB/s, p99 33.42 ms, 0 errors

Post-cleanup sanity probe from this documentation/static-check pass:

- command: python3 /private/tmp/minikv_benchmark.py --mkv build/release/mkv
  --seconds 3 --concurrency 8,16 --sizes 16,65536 --skip-ab --max-ops 300
  --preload-count 64 --workers 0 --volume-pool 8 --reuse-client-connections
- result: every scenario completed 300/300 operations with 0 errors
- c16 GET follow 64 KiB: 5.23k RPS, 326.8 MiB/s, p99 6.69 ms
- c16 PUT+DELETE 64 KiB: 1.73k ops/s, 108.4 MiB/s, p99 14.96 ms
- no minikv benchmark/test roots and no mkv/nginx/wrk processes were left
  after cleanup

Interpretation
--------------
The first bottleneck was connection churn. After keep-alive and volume pooling,
the next bottleneck is likely volume wait time, nginx/WebDAV behavior, file IO,
or metadata serialization. Raising MINIKV_VOLUME_POOL from 8 to 16 was slower
on this Mac, so the production default remains 8 and should be tuned with fd,
TIME_WAIT, p99, and volume latency metrics.

Production read
---------------
Canary is reasonable for non-critical data with full observability. Do not put
critical production data on it until these are done on production-like Linux
hardware:

- restore drill from volume files into a rebuilt LevelDB index
- rebalance drill with old and new volume sets
- sustained benchmark with keep-alive clients, fd/TIME_WAIT/RSS capture, and
  p50/p95/p99 latency
- Linux perf/flamegraph profile under read and write load
- monitoring and alerting for master 5xx, volume failures, LevelDB errors,
  disk/inode pressure, multipart scratch growth, fd count, RSS, and latency
- documented rollback and backup policy tested by operators

Next useful performance work
----------------------------
1. Add explicit backpressure for in-flight writes and replica operations.
2. Stream ordinary PUT bodies to temporary files or directly to volumes.
3. Resolve/cache configured volume addresses instead of repeating resolver
   work on hot paths.
4. Build a committed benchmark harness with JSON output and cooldown metadata.
5. Re-profile on the target Linux deployment with perf and flamegraphs.
